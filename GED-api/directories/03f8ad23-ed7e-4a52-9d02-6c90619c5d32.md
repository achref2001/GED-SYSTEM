# Pagination & Filtering Implementation Plan

## 1. System Diagnostic

### 1.1 Condor Mobility Backend (`condor-mobility-backend`)

#### Current Architecture

- **Routing Layer** (`app/api/v1/*.py`): Endpoints use `Depends(pagination_params)` to parse HTTP query strings into `skip` and `limit`. No filter parameters are extracted before passing requests to the business logic.
- **Dependency Layer** (`app/core/deps.py`): `pagination_params` provides basic skip/limit functionality but lacks integration with dynamic filters.
- **Logic Layer** (`app/logic/BaseLogic.py`): The `get_paginated` method only accepts `skip: int = 0, limit: Optional[int] = None`, completely bypassing context filters.
- **Data Access Layer** (`app/dal/BaseDAL.py`): The base `get_paginated` method executes `select(self.model)` and counts rows via `select(func.count()).select_from(self.model)` but applies zero `WHERE` clause predicates.

#### Required Structural Refactoring

- Update `BaseDAL.get_paginated` to accept `filters: dict | None = None`. If filters are provided, apply them to the SQLAlchemy statement using `.filter_by(**filters)` or advanced dynamic filters (e.g., `.ilike` for strings). Ensure the `total_result` count query also applies these filters.
- Update `BaseLogic.get_paginated` to accept and forward `filters` to `self.dal.get_paginated`.
- Standardize endpoints by creating `[Feature]FilterDTO` Pydantic models injected via `Depends()`. Convert the Pydantic instance to `dict(exclude_none=True)` before passing into logic.

---

### 1.2 Mobility Dashboard Frontend (`mobility-dashboard`)

#### Current Architecture

- **API Base Client** (`lib/api/BaseClassApi.ts`): The `list` method already accepts `params?: Record<string, any>` and passes them as HTTP query params via Axios.
- **Services** (`lib/services/*.service.ts`): Services accept simple parameter blocks (e.g., `pagination?: { page: number, perPage: number }`) but ignore generic filter definitions. For example, `bus.service.ts` converts pagination to strings but omits filters.
- **UI Components**: Components request list data via services but lack an integrated structure for filter state.

#### Required Structural Refactoring

- Update service definitions (e.g., `getBuses(pagination, filters)`) to spread the `filters` object into the HTTP request params alongside pagination values.
- Ensure UI tables and hooks correctly propagate filter dictionaries to the HTTP layer using React state.

---

## 2. Global Implementation Plan

The implementation should be partitioned **feature-by-feature** to maintain system stability.

| Phase | Scope |
|-------|-------|
| **Phase 1** — Foundation | Update base contracts: `BaseDAL.py`, `BaseLogic.py`, `deps.py`, `BaseClassApi.ts` |
| **Phase 2** — Core Entities | Auth, Users, Groups, Roles |
| **Phase 3** — Fleet | Bus, Make/Model, Devices, Depot |
| **Phase 4** — Routing | Line, Stop, Line Services |
| **Phase 5** — Ops & BI | Shift, Trips, Telemetry, Passenger, Boardings |
